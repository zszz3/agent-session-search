# 代码变更审查规范

## 审查边界

- 只审查用户通过完整绝对路径选定目录中的已暂存、未暂存和未跟踪变更；目录是仓库子目录时，不得扩展到该目录之外的无关变更。
- 默认只读取代码、配置、测试和文档，不修改文件，不执行安装、构建、测试、迁移或其他可能改变环境状态的命令。
- 每条发现都必须能由实际变更和仓库上下文支持。不要把偏好、猜测或没有触发路径的可能性写成问题。

## 严重程度

- Critical：可直接造成安全事件、数据丢失、不可逆破坏或大范围不可用。
- High：存在明确触发路径的功能错误、严重回归、权限绕过或兼容性破坏。
- Medium：边界条件错误、恢复能力不足、重要测试缺口或会影响部分用户的问题。
- Low：影响较小但具体可验证的行为、可维护性或文档问题。

## 每条发现必须包含

1. 简短、直接的问题标题。
2. 严重程度。
3. 文件路径和尽可能精确的行号或代码位置。
4. 触发条件及从变更到错误结果的完整因果链。
5. 对用户、数据、兼容性或维护工作的实际影响。
6. 最小修复方向；不要直接修改代码。

## 最终报告格式

1. `# 代码变更审查报告`
2. `## 审查范围`：说明绝对审查目录、仓库根目录、检查了哪些 staged、unstaged、untracked 变更，以及无法读取的内容。
3. `## 发现`：按 Critical、High、Medium、Low 排序；同一根因只保留一条。
4. `## 验证建议`：只列与具体风险对应的最小验证命令或场景，明确说明这些命令尚未执行。
5. `## 结论`：给出“存在阻断问题”“建议修复后合并”或“未发现确定性问题”等证据一致的结论。

没有发现时，明确写“未发现确定性问题”，并说明静态审查范围和未实际运行的验证。不要为了让报告显得完整而制造问题，也不要输出无证据的表扬或笼统建议。
