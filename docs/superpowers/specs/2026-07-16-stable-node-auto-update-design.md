# 稳定 Node 自动更新设计

## 目标与范围

- 修复 npm 正式版从 App 内更新时，Electron 在覆盖自身安装目录后读取到无效 `default_app.asar` 的问题。
- 保留现有 Release 检查、下载、SHA-256 校验、全局 npm 安装、失败回滚、状态提示和自动重启流程。
- 终端安装命令继续作为失败兜底，不把“打开终端并自动输入命令”作为正常更新路径。

## 方案比较

- 采用：使用 npm 启动器记录的、位于待更新包之外的 Node 可执行文件启动独立更新器。改动集中，更新过程保持后台完成，也适用于当前 macOS 和 Windows npm 安装模型。
- 不采用：打开用户终端并自动运行 npm 命令。它需要处理终端选择、Shell quoting、窗口生命周期和 Windows 差异，用户体验也更打扰。
- 不采用：额外安装一个永久的系统级 updater。它能彻底隔离更新器，但增加安装、卸载和版本管理成本，超出当前需求。

## 执行流程

1. npm 启动器继续通过 `AGENT_RECALL_NODE_PATH` 把稳定 Node 路径传给 Electron App。
2. App 发起更新时，独立更新器必须使用该 Node 路径启动；不得回退到当前 Electron 的 `process.execPath`。
3. 更新器启动环境移除 `ELECTRON_RUN_AS_NODE`，等待 App 主进程退出后执行现有安装与校验流程。
4. 缺少或无法启动稳定 Node 时，在覆盖安装开始前失败，沿用现有原生错误弹窗和手动 npm 命令。
5. 安装成功后继续通过全局 `agent-recall` 命令重新启动 App。

## 错误处理与安全

- 不信任来自会话内容或 UI 的可执行路径；仅使用 npm 启动器写入的环境值或测试显式注入值。
- 不在 Node 启动失败时再次尝试包内 Electron，避免重新触发自覆盖。
- 下载校验、npm registry、安装锁、旧包备份和回滚语义保持不变。
- 安装与升级测试只使用临时 HOME、临时 npm prefix 和合成包，不写真实全局安装。

## 验证

- 单元测试证明 GUI 更新器选择稳定 Node，并且子进程环境不含 `ELECTRON_RUN_AS_NODE`。
- 回归测试证明缺少稳定 Node 时在安装前给出明确错误。
- 临时 npm prefix 冒烟测试覆盖旧版到新版的完整安装，且结束后清理临时包与进程。
