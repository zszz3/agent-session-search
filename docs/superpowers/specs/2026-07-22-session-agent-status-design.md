# Session Agent 状态分析设计

## 背景与目标

AgentRecall 已经索引会话消息、工具调用、工具结果、错误、上下文压缩事件和会话元数据，但 Session 详情仍主要展示原始轨迹。长会话中，用户需要自己翻阅大量内容，才能判断 Agent 当前停在哪里、调用过哪些工具、是否连续失败，以及显式计划完成到了哪一步。

本次新增一个由用户手动触发的只读分析入口。用户打开 Session 详情后点击“分析状态”，应用从完整的已索引轨迹中提炼一份紧凑的 Agent 状态卡。分析只用于当前界面展示，不写回原会话、不保存到数据库、不注入 Agent，也不调用大模型。

## 已确认方案

采用确定性的本地代码分析：

- 普通 Session 详情由主进程读取完整轨迹并返回结构化结果，避免详情分页只加载最近消息时少算工具调用。
- 远程详情快照直接使用快照中已有的完整消息与轨迹执行同一套纯函数分析。
- 工具调用按 `callId` 去重；同一次调用的发起事件与结束事件只计一次。
- 只展示有明确轨迹证据的事实。TODO 仅从结构化计划工具中提取，不从自然语言猜测；缺少证据的字段显示为数据不足或直接省略。

不采用只分析当前页面的方案，因为详情默认只加载最近一页，统计会不完整；也不采用大模型总结，因为计数与状态归纳需要可重复、低延迟且不产生幻觉的结果。

## 状态数据模型

新增只面向展示的 `SessionAgentStatus` 结构，包含：

- `state`：`running`、`waiting_agent`、`waiting_user`、`failed`、`interrupted` 或 `unknown`。
- `latestUserRequest`：最近一条有效用户消息的紧凑预览。
- `todos`：最近一次结构化计划中的条目及其 `pending`、`in_progress`、`completed`、`cancelled` 状态。
- `toolCalls`：去重后的调用总数，以及按工具名称聚合的次数、失败次数和未知结果次数。
- `failureCount` 与 `latestFailure`：有明确失败状态的调用或错误事件数量，以及最近一次失败的简短描述。
- `compactionCount` 与 `abortedCount`：上下文压缩和中断事件次数。
- `projectPath`、`firstActivityAt`、`lastActivityAt`：可直接观测的环境和时间信息。
- `messageCount`、`traceEventCount`、`analyzedAt`：说明本次分析覆盖范围与生成时间。

状态采用保守规则：

1. 会话被实时检测为 Open 时标记为 `running`。
2. 最新轨迹是明确错误或中断时分别标记为 `failed` 或 `interrupted`。
3. 已关闭会话最后一条可见消息来自用户时标记为 `waiting_agent`；来自助手时标记为 `waiting_user`。
4. 没有足够证据时标记为 `unknown`，不推断“已完成”。

## 工具与事件归纳

分析器以轨迹事件为事实源：

- `tool_result` 不单独计为一次工具调用。
- 具有相同 `callId` 的 `tool_call` 与结束 `event` 合并；名称优先使用调用事件的标题，并去掉标题中参数摘要部分。
- 没有 `callId` 的调用事件按自身索引独立计数。
- `status: failure` 与 `eventType: error` 计入失败；未知结果保持未知，不按成功处理。
- `context_compacted`、`turn_aborted` 和普通错误属于状态事件，不混入工具排行榜。
- `update_plan`、`TodoWrite`、`rewrite_todo_list` 等明确的结构化计划载荷可用于 TODO；无法解析或仅出现在普通对话中的清单不进入状态卡。

自然语言预览会去除多余空白并限制长度。失败详情只展示紧凑片段，不在状态卡中展开完整工具输出。

## 数据流与生命周期

本地或 SSH Session：

1. 用户打开详情，原有分页加载行为不变。
2. 点击“分析状态”后，渲染进程调用新的 `session:agent-status` 接口，并传入当前 Open/Closed 观测。
3. 主进程确保远程详情已按现有机制就绪，读取 Session 元数据、最近消息和完整轨迹，调用纯分析函数。
4. 渲染进程只保存当前详情组件内的临时结果并展示状态卡。

远程快照详情：

1. 点击同一入口。
2. 使用快照已有的完整消息和轨迹调用同一纯分析函数。
3. 不发起额外同步，也不修改远程数据。

关闭详情会丢弃结果。会话继续运行后，旧卡片保留“分析于”时间，用户可点击“重新分析”主动刷新；本次不增加轮询或自动重算。

## Session 详情交互

- 在现有详情操作区的摘要相关操作旁增加“分析状态”按钮，使用与现有操作一致的尺寸和颜色，不引入新的主按钮层级。
- 分析期间按钮显示“分析中…”并防止重复触发。
- 成功后在操作区与会话正文之间显示紧凑的“Agent 状态”卡；再次点击按钮会覆盖旧结果。
- 只读远程详情仍显示这一分析入口，但不解锁重命名、迁移、删除等写操作。
- 失败时在卡片位置显示简短错误与重试入口，不影响原会话内容浏览。

状态卡采用现有主题变量和信息密度：

- 标题行展示当前状态徽标、覆盖消息/轨迹数量和分析时间。
- 最近用户诉求占一行宽区域。
- TODO、工具调用、失败、压缩/中断等使用紧凑键值布局；工具 Top 项使用小型标签显示次数。
- TODO 数量较多时仅展示当前进行中、未完成以及少量最近完成项，并标明总数，避免卡片挤占会话正文。
- 没有工具或 TODO 时显示简短的“未发现明确记录”，不渲染空白网格。

## 错误与边界情况

- Session 已被删除或远程详情无法加载时返回明确错误，详情保持可用。
- 没有消息但存在轨迹时仍可展示工具和错误统计；两者都为空时显示数据不足。
- 时间戳无效时不参与首末活动时间计算，不阻断其他字段。
- 极长会话只在用户点击时分析。普通打开详情、分页和搜索路径不增加额外工作。
- 分析结果完全来自用户本机或已打开的远程会话内容，不上传到新的服务。

## 测试与验证

- 分析器单元测试覆盖调用去重、无 `callId` 事件、失败与未知状态、压缩/中断、状态判断、最新用户诉求和结构化 TODO。
- IPC 测试确认分析读取完整轨迹，而不是详情当前页，并能处理缺失 Session。
- Session 详情测试覆盖手动触发、加载态、结果展示、重新分析、失败重试和只读远程入口。
- 执行针对性测试、完整测试、生产构建与 `npm run release-note:check`。
- 在 Electron 中手动检查普通 Session、Open Session、长会话和只读远程详情的布局与交互；验证后停止测试进程，不遗留临时数据或安装产物。

## 非目标

- 不把状态栏注入 Codex、Claude Code 或其他 Agent。
- 不持久化分析结果，也不增加历史版本。
- 不用大模型生成状态、补全 TODO 或判断任务是否真正完成。
- 不自动限制工具调用或改变 Resume 行为。
- 不替换现有 AI 摘要、全文轨迹或会话搜索。
