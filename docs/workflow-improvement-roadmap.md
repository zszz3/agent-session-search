# Workflow 后续改进路线

## 文档目的

本文基于 AgentRecall 2.0 当前 Workflow 实现，梳理已经具备的能力、仍然存在的产品与工程缺口，并给出按优先级推进的后续路线。

本路线不以继续扩张执行内核为第一目标。当前 Workflow 已具备 DAG 并行调度、人工输入、脚本审批、节点暂停与重试、运行中修订、状态持久化恢复和多 Runtime 执行等基础能力。下一阶段应优先让用户能够理解一次运行发生了什么、复用已有 Workflow、从外部安全控制完整生命周期，并持续评估运行质量。

## 当前能力判断

### 已经形成的执行底座

- Workflow V2 使用冻结后的定义和计划执行，能够校验拓扑、输入输出合同及上下文预算。
- 调度器支持并行节点、最大并发数和资源锁，不要求所有 Workflow 都退化为线性执行。
- LLM 节点支持重试、进度探测、超时监督、暂停和人工介入。
- Script 节点支持结构化输入、能力分析、风险分级、一次性审批和执行超时。
- 用户能够在节点执行期间对 Agent 会话补充消息、确认或驳回完成提议。
- 暂停、失败或停止后的 Run 支持基于新图版本修订并继续执行。
- Workflow、Run、节点会话、输出和恢复控制状态能够持久化，Renderer 重载不会创建第二套执行引擎。
- Runtime、MCP、Workflow 和宿主应用之间已经形成相对清晰的边界。

### 当前主要矛盾

当前问题已经不再是“Workflow 能不能运行”，而是以下能力尚未形成完整产品闭环：

1. 用户难以回看、比较和解释多次运行。
2. MCP 可以辅助规划 Workflow，但无法控制完整运行生命周期。
3. 官方模板和个人 Workflow 缺少复制、导入、导出与依赖迁移能力。
4. 日常编辑仍需要理解较多底层定义结构。
5. 宿主目录与 Automation Engine 中存在两套可能漂移的 Workflow shared 合同。
6. Evaluation 与 Workflow Run 尚未形成直接反馈循环。

## 优先级一：Workflow Run Center

这是最值得优先落地的改进。现有界面主要围绕 Workflow 本身和当前 Run 展示，用户不能方便地选择任意历史 Run，也缺少跨 Run 对比所需的统一视图。

### 用户能力

- 在 Workflow 详情中查看完整 Run 列表，而不只显示正在运行或最近失败、停止的 Run。
- 按状态、触发方式、时间范围和 Revision 筛选 Run。
- 查看每次 Run 的开始时间、结束时间、总耗时、触发来源和最终结果。
- 查看节点时间线，包括排队、执行、等待人工输入、审批、重试和完成阶段。
- 查看每个节点实际使用的 Runtime、Channel、模型、attempt、Token、成本和耗时。
- 查看节点输入、输出摘要、错误、人工决策和产物引用。
- 对比两个 Run 或两个 Revision 的拓扑、配置、节点结果、耗时、成本和验收结果。
- 从失败节点重新运行，并明确展示复用了哪些上游结果、重新执行了哪些节点。

### 状态合同建议

Run 和节点观测数据应采用稳定、可持久化的合同，至少补充：

- Run 触发来源：手动、定时、MCP、恢复或重跑。
- Runtime、Channel、模型和配置快照，避免当前配置变化后无法解释历史结果。
- 节点排队、开始、暂停、恢复和结束时间。
- attempt 级别的错误、输出摘要、Token 和成本。
- 事件 sequence，保证重载、恢复和重复订阅时能够稳定去重。
- 输出摘要或 digest，支持判断缓存与重跑是否真的复用了相同结果。
- 等待用户、等待审批和实际执行的分段耗时。

### 交互建议

- Workflow 页面增加 `Runs` 入口，与定义、画布和输出保持同级。
- Run 列表选择历史记录后，画布切换为该 Run 的只读执行快照。
- 时间线与画布使用相同节点状态颜色，但不依赖颜色表达唯一含义。
- 对失败节点提供“查看错误”“从此处重跑”“基于当前定义创建新 Run”三个明确动作。
- 历史 Run 默认只读，任何继续执行或重跑都创建可审计的新动作记录。

### 验收标准

- 用户可以从同一个 Workflow 打开任意历史 Run。
- 应用重启后仍能查看完整节点时间线和运行配置快照。
- 并行节点在时间线上能够体现实际重叠区间。
- 用户能够区分执行耗时、等待输入耗时和等待审批耗时。
- 从失败节点重跑时，不会静默重复执行存在副作用的 Script 节点。

## 优先级二：补齐 Workflow MCP 生命周期

当前 MCP 更偏向 Workflow 规划与上下文写入。若 AgentRecall 希望成为真正可编排的自动化平台，需要让可信客户端在同一个受控接口中完成确认、执行、等待、回答和结果读取。

> 实施设计见 [Workflow MCP 生命周期与节点工具可观测性设计](./workflow-mcp-lifecycle-design.md)。本阶段同时统一节点 Runtime 的 MCP 上下文注入，并把工具调用与结果纳入节点会话主时间线。

当前 V1 已完成受控读写授权、生命周期查询与变更工具、Codex/Claude/ACP Runtime 节点上下文注入、结构化节点完成身份与合同校验，以及节点会话工具时间线。Codex、Claude Code 与 ACP Runtime 现已共用 planning/node execution 能力策略，OpenCode、Hermes、OpenClaw 通过 ACP 继承同一规则；规划会话会持久化并显示 MCP 调用与失败原因。生命周期边界已补充稳定错误码、筛选校验、重复 intervention 冲突处理；输出查询提供不含绝对路径的类型、大小和受限脱敏预览。远程客户端写授权、Run 对比与失败节点重跑仍属于后续范围。

### 建议新增工具

- `workflow_confirm`：确认指定 Revision，要求 expected revision。
- `workflow_run`：启动 Run，返回 workflowId 和 runId。
- `workflow_run_list`：按 Workflow、状态和时间列出 Run。
- `workflow_run_get`：返回 Run、节点状态、待处理人工动作和输出摘要。
- `workflow_stop`：停止指定 Run，要求 runId 精确匹配。
- `workflow_intervention_resolve`：处理人工介入、Reviewer 决策和恢复动作。
- `workflow_script_input_submit`：提交符合节点参数合同的结构化输入。
- `workflow_outputs_list`：列出 Run 产物及安全的预览信息。

### 安全边界

- 读取工具与状态变更工具分开定义，避免一个模糊工具同时承担查询和执行。
- 任何确认、停止、继续和输入提交都必须携带 workflowId、runId 或 expected revision 等并发保护字段。
- Script 副作用审批不能因为调用来自 MCP 就被绕过。
- 返回错误使用稳定 code 与用户可读 message，避免客户端解析自由文本。
- 默认不返回密钥、完整环境变量、授权头或本地敏感路径。
- MCP 断开不应停止已经被主进程接管的 Run，也不应留下无法处理的 approval waiter。

### 验收标准

- MCP 客户端可以完成“创建草稿—校验—确认—运行—处理输入—读取结果”的完整闭环。
- 对过期 revision、错误 runId 和已解决 intervention 的重复提交能够稳定拒绝。
- MCP 与桌面端同时观察同一 Run 时不会产生双重执行或重复回答。

## 优先级三：Workflow 复用与分发

官方 Workflow 和个人 Workflow 已经有清晰分类，但还需要形成可迁移的 Workflow 资产模型。

### 用户能力

- 从官方 Workflow 创建个人副本，官方模板本身保持只读。
- 复制个人 Workflow，并选择是否保留长期上下文。
- 导出单个 Workflow 或 Workflow 包。
- 从文件导入 Workflow，在保存前完成结构和依赖校验。
- 显示所需 Runtime、模型、Skill、MCP Server、输入文件和权限能力。
- 导入后为缺失依赖提供映射和修复入口。

### 可移植格式

导出格式应包含：

- 格式版本、Workflow 标识、标题、目标和图定义。
- 节点 Runtime 能力要求，而不是强绑定另一台设备的本地 Channel ID。
- 可选的模板资源、说明文档和示例输入。
- 依赖清单及最低兼容版本。
- 不包含历史 Run、密钥、授权数据和设备绝对路径。

### 验收标准

- 官方模板可以一键复制为可编辑的个人 Workflow。
- Workflow 包能够在 macOS 与 Windows 之间迁移，不沿用原机器绝对路径。
- 导入缺失依赖的 Workflow 时可以保存为不可运行草稿，并明确展示修复项。

## 优先级四：降低 Workflow 编辑门槛

JSON 定义编辑适合高级用户和调试，但不应成为日常调整节点的主要方式。

### 建议交互

- 在画布中新增、连接、复制和删除节点。
- 使用节点表单配置输入、输出、Runtime、模型、重试、超时和资源锁。
- 在边上展示字段映射和条件，保存时检查来源字段是否存在。
- 即时展示拓扑错误、缺失依赖、类型冲突和不安全脚本能力。
- 提供模板节点与常用组合，而不是要求用户从空白 JSON 开始。
- 保留 JSON 高级模式，并在切换时保证结构化编辑与源码一致。

### 设计原则

- 错误尽量显示在对应节点或边上，不只在全局 Toast 中展示。
- 草稿保存、独立 Review 和用户确认继续保持三个不同阶段。
- 运行中的图修订必须明确显示基于哪个 graph version，以及哪些节点会重新执行。

## 优先级五：统一 Workflow shared 合同

当前宿主目录与 Automation Engine 下同时存在 Workflow shared 文件，其中部分文件仍然一致，部分文件已经出现内容差异。这会造成同名类型在不同边界表达不同能力的风险。

### 建议方案

优先选择一个唯一真源：

1. 由 Automation Engine 的 shared 目录统一提供合同。
2. 宿主、Preload、MCP 和 Renderer 均从稳定的 Engine 导出入口引用。
3. 删除不再被使用的宿主镜像文件。

如果打包或隔离边界必须保留两份文件，则应：

- 通过脚本从唯一真源生成镜像，禁止手工维护。
- 在 CI 中增加导出符号和内容一致性检查。
- 为合同增加 schema version，并为持久化迁移提供兼容策略。

### 验收标准

- 同一个 Workflow 类型只有一个人工维护的定义来源。
- IPC、MCP、持久化和 Renderer 不再分别推断相同字段。
- 合同变化会通过类型检查和 schema 兼容测试同时被发现。

## 优先级六：Workflow 与 Evaluation 闭环

Evaluation 不应只是独立页面，也应成为 Workflow Revision 的质量反馈。

### 建议能力

- Run 完成后按验收标准生成评估结果。
- 展示节点成功率、重试率、人工驳回率、总耗时和成本趋势。
- 比较不同 Revision 在固定样例集上的结果。
- 将表现稳定的历史 Run 固化为回归样例。
- 新 Revision 发布前可选择运行回归集，并明确展示退化项。

### 验收标准

- 用户能够回答“这次修改是否比上一个 Revision 更好”。
- 评估结论可以追溯到具体 Run、节点输出和评估器版本。
- 评估失败不会篡改或覆盖原始 Workflow 输出。

## 推荐实施顺序

### 阶段一：运行可观测底板

- 扩展 Run、节点 attempt 和事件观测合同。
- 持久化 Runtime 配置快照、耗时、Token、成本和触发来源。
- 增加历史 Run 查询与详情读取接口。
- 先实现只读 Run 列表、节点时间线和错误详情。

### 阶段二：恢复与重跑

- 增加失败节点重跑计划预览。
- 明确缓存复用和 Script 副作用策略。
- 支持 Run 与 Revision 对比。
- 增加应用崩溃、Runtime 断开和恢复过程的故障注入测试。

### 阶段三：外部控制闭环

- 补齐 Workflow MCP 生命周期工具。
- 对桌面端和 MCP 并发操作增加 revision/run identity 保护。
- 建立稳定错误码和敏感字段过滤测试。

### 阶段四：资产化与低代码编辑

- 增加官方模板复制、个人 Workflow 复制和导入导出。
- 增加依赖描述与跨设备映射。
- 把日常 JSON 修改逐步迁移为画布和表单操作。

### 阶段五：质量回归

- 将 Evaluation 结果关联到 Workflow Run 与 Revision。
- 建立固定样例集、趋势对比和发布前回归入口。

## 本路线暂不建议优先做的事项

- 不优先增加更多 Runtime 类型；现有 Runtime 数量已足以验证抽象。
- 不优先继续增加官方 Workflow 数量；先解决复制、迁移和依赖检查。
- 不优先引入新的节点执行模型；先把 LLM、Script 和人工介入的产品闭环做完整。
- 不把 Run 历史简单实现为日志文本列表；必须能够关联节点、attempt、配置和结果。
- 不允许“从失败节点重跑”静默绕过 Script 审批或重复执行副作用。

## 下一项推荐工作

下一项独立开发任务建议定为：

> Workflow Run Center V1：支持历史 Run 选择、节点时间线、运行配置快照、耗时与错误详情，并为后续 Token、成本、Run 对比和失败节点重跑建立稳定合同。

V1 暂不包含重跑和完整成本统计，但数据模型不应阻塞这些后续能力。
